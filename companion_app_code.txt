How the Application Works

Your two-tier architecture runs a real application with a frontend and backend that communicate
through the network:


 Component                       Description
 Frontend (Public Tier)          Nginx serves an HTML page. When the page loads,
                                 JavaScript calls /api/health and /api/data. Nginx
                                 reverse-proxies any request starting with /api/ to the
                                 backend.
 Backend (Private Tier)          A Node.js Express server running on port 3000. It exposes
                                 two endpoints: GET /api/health and GET /api/data. It is NOT
                                 accessible from the internet — only from the frontend tier.
 Reverse Proxy                   Nginx on the frontend catches /api/* requests and forwards
                                 them to the backend’s private IP on port 3000. The browser
                                 never talks to the backend directly.

Backend: Node.js Express API

This is a single file: server.js. It runs on port 3000 and exposes two endpoints.



server.js

  const express = require("express");
  const { execSync } = require("child_process");

  const app = express();
  const PORT = 3000;

  // Helper: get instance metadata (IMDSv2)
  function getMeta(path) {
    try {
      const token = execSync(
        'curl -s -X PUT "http://169.254.169.254/latest/api/token" ' +
        '-H "X-aws-ec2-metadata-token-ttl-seconds: 21600"'
      ).toString().trim();
      return execSync(
        `curl -s -H "X-aws-ec2-metadata-token: ${token}" ` +

# ─── Configure Nginx reverse proxy ───
# Replace BACKEND_IP below with one of your backend instance's private IP
cat > /etc/nginx/conf.d/reverse-proxy.conf << 'NGINXEOF'
server {
    listen 80;
    server_name _;

     root /usr/share/nginx/html;
     index index.html;

     location / {
         try_files $uri $uri/ =404;
     }

     location /api/ {
         proxy_pass http://BACKEND_IP:3000;
         proxy_set_header Host $host;
         proxy_set_header X-Real-IP $remote_addr;
         proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
     }
}
NGINXEOF

# Remove the default server block to avoid conflicts
rm -f /etc/nginx/conf.d/default.conf
sed -i '/^\s*server\s*{/,/^\s*}/d' /etc/nginx/nginx.conf 2>/dev/null || true

# Start Nginx
systemctl start nginx
systemctl enable nginx



⚠️ Warning
You must replace BACKEND_IP in the script with the actual private IP of one of your
backend instances.
You can either hardcode it after your first deploy, or use a Terraform variable to inject it
dynamically.


⭐ Bonus
Instead of hardcoding a single backend IP, create an Internal ALB in your private subnets
that load balances across your backend ASG instances.
Then replace BACKEND_IP:3000 with your Internal ALB’s DNS name:80 in the Nginx config.
This is how production two-tier architectures work — the frontend never talks to individual
backend instances directly.

How to Verify It Works

After deploying with Terraform, follow these steps to verify the full flow:



1. Test the Frontend via ALB

Open your browser and go to the External ALB DNS name. You should see the Two-Tier App
page with your name. The health check card should show "healthy" with the backend instance
ID, and the data card should display the table of AWS services.



2. Test the Backend Directly (from Bastion)

SSH into your bastion host, then SSH into a backend instance and test the API locally:


  # From your machine → bastion
  ssh -i your-key.pem ec2-user@<BASTION-PUBLIC-IP>

  # From bastion → backend instance
  ssh -i your-key.pem ec2-user@<BACKEND-PRIVATE-IP>

  # On the backend instance, test the API
  curl http://localhost:3000/api/health
  curl http://localhost:3000/api/data


You should see JSON responses with the instance ID and availability zone.



3. Confirm the Backend is NOT Public

Try accessing a backend instance’s private IP directly from your browser or local machine. It
should NOT work — the backend is only reachable through the Nginx reverse proxy or via the
bastion. This confirms your security group and subnet configuration is correct.

 📝 Note
 The user data scripts contain the entire application. You do not need separate app files —
 everything is embedded in the shell scripts and gets created on the EC2 instances at boot
 time.
 The index.html and server.js sections above are shown separately for readability, but they
 are already included inside the user data scripts.




Important Notes

   •​   Replace your name: Change [YOUR NAME HERE] in the web.sh user data script
        before deploying.

   •​   Replace BACKEND_IP: After your first terraform apply, get a backend instance’s private
        IP from the console or outputs, update web.sh, and re-deploy the web tier.

   •​   Backend SG port: Your Backend-SG must allow inbound TCP port 3000 from Web-SG
        (not port 80). The Express server listens on 3000.

   •​   NAT Gateway required: The backend instances need internet access (via NAT
        Gateway) to install Node.js and npm packages during boot.

   •​   Boot time: The backend takes 1–2 minutes to boot because it installs Node.js and npm
        packages. If the frontend shows "unreachable" at first, wait and refresh.
